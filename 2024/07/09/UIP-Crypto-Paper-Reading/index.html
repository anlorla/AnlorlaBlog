Information

  • title: End-to-Same-End Encryption: Modularly Augmenting an App with an Efficient, Portable, and Blind Cloud Storage
  • author: Long Chen, Institute of Software, Chinese Academy of Sciences; Ya-Nan Li and Qiang Tang, The University of Sydney; Moti Yung, Google & Columbia University
  • year: 2022

Outset

Intro

With cloud services becoming more and more prevalent, how can we ensure that data on cloud services is secure? Existing cloud storage services either store user data directly or hold decryption keys, which means that servers have access to plaintext data and there is a risk of privacy leakage. As a result, the paper proposes a new end-to-end encrypted cloud storage system, Portable Blind Cloud Storage (PBCS), to ensure that the data remains encrypted during transmission and storage, and is completely invisible to the cloud server, as seen by the ‘B’ (Blind) in PBCS is important!

Requirement

In the class, Mr. Tang explained to us that the application of cryptography always needs to consider security and usability, balancing the compromise between the two, and only the system architecture that follows these two properties has the chance to come into the public’s view.

In addition to this, it is often necessary to consider the following indicators:

  • Portability & cloud-blindness

    Portability is a desired prop erty since a user may access his content from multiple devices, such as mobile phones, laptops, and desktops. To protect user data against cloud, the data stored in cloud should be blind to the cloud.

    Password-hardened encryption(PHE) is one of the cryptographic tools to strengthen the security of password protected storage. It is secure against external attackers that may breach the storage server.

  • Deployability

    It can support deployment with access to more APIs than just storage, retrieval, deletion and access control.

  • Built-in App login

    The problem of compromised passwords that may result from the frequent reuse of passwords by users. When the App server is corrupted, the login password may be learnt via offline attacks. Then the adversary can use this password to breach into the secure storage system

Architecture Overview

First of all, we can overview the design of the whole system from a global perspective, one involves three entities: Key Server, Client and Data Server, the main role of the three are as follows:

  • Client interact with the key server to generate high-entropy ciphers for authentication and share master keys for data encryption before storing them with the data server .
  • The key server collaborates with the data server by providing authentication ciphers and managing encryption keys to ensure secure storage and access to data in a cloud environment.
  • The data server receives the encrypted data from the client, stores it securely, and allows authorized access based on the authentication provided by the key server, thus creating a secure data management system in the cloud storage architecture.

The interaction process between the three entities and the data flow is shown below.

The process can be described as follows:

  1. the client derives his password by querying the key server with his passphrase.
  2. the client logins the data server with his ID and the derived password.
  3. the client logins the key server.
  4. the client “shares” his master key.
  5. the client encrypts his data with the master key and deposit the ciphertext to the data server.

Main system architecture/technology

Based on the above, we can further understand the main architecture of the system. It’s worth mentioning that PBCS fully leverages the existing infrastructure of a typical App.

The PBCS system has two parts: KEM(Key Encapsulation Mechanism) for generating and storing a strong master key and DEM(Data Encapsulation Mechanism) for encrypting content, with the security of the master key determining the confidentiality and integrity of the data.

prior knowledge/defination

IND-CPA: INDistinguishability under Chosen Plaintext Attack, golden standard to evaluate the security of a system.

End-to-End Encryption: According to Mr. Tang Qiang’s description in class, the main encryption methods we discussed are End2End and Hop2Hop, the difference between the two is that End2End encryption occurs only at the communication endpoints (i.e., the sender and intended recipient) for encryption and decryption to occur. In this setup, no intermediary party or service provider, such as an Internet Service Provider (ISP) or a messaging platform, has access to plaintext data, even if they are responsible for transmitting it. The data is encrypted on the client side and can only be decrypted by the user with the correct key,the cloud server does not have access to the plaintext data.

Simple cloud server authtication

With IBOPRF (Identity Based Oblivious PRF), the client only needs to memorize a short passphrase to generate a long passphrase, and the pseudorandomness ensures that the low entropy passphrase remains secret from the (malicious) data server when the (malicious) data server blindly checks the IBOPRF a finite number of times with the same identity.

In practice, users often reuse their passwords, or use different passwords for different services, which can lead to key leakage to the data server.To solve this problem, one idea is to let the application login complete naturally in thr meantime of that the “Give” or “Take” protocol is executed.

Evaluations

Deployability

PBCS system is easy to be adopted to existing Apps, simply run the keyServerAPI on the App provider’s administrative server and call the clientAPI on the App’s client side to provide secure deposit and retrieve.

Efficiency

The implementation of cryptographic primitives using standard Java API and Bouncy Castle library ensures efficiency in the PBCS system, with optimizations like TLS 1.2 and AES encryption for secure communication and storage.

Economical Cost

Real-world experiments conducted on Amazon Web Service (AWS) highlight the cost-effectiveness of the PBCS system, making it an economically viable solution for secure cloud storage deployments.

Scalability

Through throughput tests on key server performance, the scalability of the PBCS system is evaluated, showcasing its ability to handle increasing requests and data volumes efficiently

Advantages and disadvantages

  1. Advantages

    1. Enhanced privacy protection: Data is completely invisible to the cloud server, solving the privacy issues in existing cloud storage services.
    2. Ease of use: User-friendly experience, only one password phrase is needed to access the data without complex key management.
    3. Efficient performance: Experimental results show that the system operates efficiently in real network environments.
    4. Flexibility and Portability: Users can access their data on any device without worrying about data security.

    Disadvantages

    1. Key management is user-dependent: Users need to keep their password phrases safe and may not be able to access their data if they lose their password phrases.
    2. Complexity: The system design and implementation are relatively complex, especially in key management and data encryption and decryption.
    3. Performance overhead: although experiments have shown the system to be efficient, encryption and decryption operations still introduce some performance overhead, especially when dealing with large-scale data.
    4. Reliance on external key servers: The system relies on key servers for key management, and if the key servers are compromised, the security of the system may be compromised.

Extended applications

Instantiating PBCS in different settings could lead to different interesting new applications. The following three examples are scenarios where the PBCS system is applied:

  • Secure personal repository

    Keeping personal multimedia data such as pictures and video albums secure in cloud and available across devices is a general usage of cloud storage. PBCS in this paper has significantly improved the preliminary version both on the usability and security. Particularly, it allow the data server to be plain cloud storage, and reuses the App log-in requiring only one single password after upgrade.

  • Secure sync-up system for E2EE messaging Apps

    PBCS can help users synchronize their contact lists and chat history in an encrypted manner on a third-party cloud server chosen by themselves, while only use the App provider’s server as the key management server of PBCS. Therefore the contact list can only be accessed by the user with correct passphrase.

  • Portable personal AI assistant with privacy guarantee

    Once a personal model is generated, the user can securely store her personal AI assistant model in the cloud, and get access to it via any other device or when changing devices. With PBCS, the user plain data never leaves the user devices and the user has full control of his own data and personal AI assistant model which remains private when passing across devices.

Lessons Learned

As mentioned in Occam’s razor principle: “simpler solutions are more likely to be correct than complex ones”, the simpler the architecture of a system, the better, and we need to move towards high compatibility and low redundancy in our architectural design. We need to approach high compatibility and low redundancy in our architectural design, and at the same time pay attention to simplify engineering and reasonable maintenance.
